# API Tech Debt

> Generated: May 9, 2026 | Branch: development | Commit: 07478fe

## Overview

This document catalogs known technical debt, high-coupling zones, risky areas, and test gaps in the API. Items are prioritized by impact on reliability and maintainability. Many were discovered during implementation and logged in ADRs or inline TODOs; this serves as a consolidated snapshot.

## High-priority debt

### 1. Intent Classifier Dependency (Impact: HIGH)

**Location:** `services/intent-classifier.ts` → Cloudflare AI Gateway

**Issue:** All intent classification depends on a single external API (Cloudflare). Latency, rate limits, or outages directly impact user experience.

**Evidence:**
- No fallback classifier
- No local classification model
- No intent cache

**Risk:** If Cloudflare is down, users can't interact with the bot.

**Recommendations:**
- [ ] Implement local fallback classifier (heuristic or small model)
- [ ] Cache intent classifications for 1 hour
- [ ] Add circuit breaker pattern
- [ ] Monitor classifier latency SLA

---

### 2. Test Coverage (Impact: HIGH)

**Location:** `src/tests/` — Only 1-2 test files, primarily prompt validation

**Issue:** Core business logic (orchestrator, kernel, enrichment) has minimal test coverage.

**Evidence:**
- `intent-classifier.test.ts` — basic tests
- `prompt-architecture-guard.test.ts` — schema validation only
- Zero tests for: `agent-orchestrator.ts`, `model-turn-runner.ts`, `tool-executor.ts`

**Risk:** Regressions go undetected. Refactoring is dangerous.

**Recommendations:**
- [ ] Add vitest fixtures for `ModelTurnRunner` and `ToolRegistry`
- [ ] Mock external APIs (TMDB, YouTube, Cloudflare)
- [ ] Test kernel loop with synthetic LLM responses
- [ ] Add integration tests with test database
- [ ] Target 70%+ coverage for core modules

---

### 3. Multi-provider Fallback Logic (Impact: MEDIUM)

**Location:** `core/model/credential-pool.ts`

**Issue:** Fallback between providers is implicit and not well-tested.

**Current logic:**
```ts
const provider = credentialPool.getForModel(modelName);
// If model not registered, uses default (Gemini)
// If Gemini fails, no retry to Claude
```

**Risk:** LLM failures cascade; user sees error instead of retry.

**Recommendations:**
- [ ] Implement explicit retry loop (try Cloudflare → Gemini → Claude)
- [ ] Add exponential backoff + jitter
- [ ] Log provider fallback for observability
- [ ] Test provider failure scenarios

---

### 4. Conversation Context JSONB Mutations (Impact: MEDIUM)

**Location:** `services/conversation-service.ts`, throughout routes

**Issue:** JSONB context is mutable; easy to accidentally drop fields or create corrupted state.

**Example problem:**
```ts
// Unsafe—spreads context, but nested mutations not tracked
await db.update(conversations)
  .set({ context: { ...oldContext, memoryIds: newIds } })
  .where(eq(conversations.id, id));
```

**Risk:** Silent data corruption if nested mutations occur.

**Recommendations:**
- [ ] Create strongly-typed context builder class
- [ ] Validate context shape on every update (Zod schema)
- [ ] Use immutable data structures (Immer) or Object.freeze()
- [ ] Add migration warnings if context is malformed

## Medium-priority debt

### 5. Webhook Idempotency (Impact: MEDIUM)

**Location:** `routes/webhook/telegram.ts`, `routes/webhook/whatsapp.ts`

**Issue:** Webhook handlers don't validate idempotency keys. Duplicate webhooks cause double processing.

**Risk:** User saves same memory twice if webhook retries.

**Recommendations:**
- [ ] Add `idempotency_key` to message records (indexed)
- [ ] Deduplicate incoming messages before queueing
- [ ] Return 200 for duplicate webhooks (idempotent response)

---

### 6. Embedding Sync & Invalidation (Impact: MEDIUM)

**Location:** `core/memory/embedding-service.ts`

**Issue:** Embeddings generated by Cloudflare may drift from cache. No scheduled sync.

**Risk:** Semantic search results become stale if embedding updates lag.

**Recommendations:**
- [ ] Add nightly embedding regeneration for updated items
- [ ] Store embedding `generatedAt` timestamp
- [ ] Warn if embedding is >30 days old
- [ ] Test embedding staleness impact on search

---

### 7. Bull Queue Observability (Impact: MEDIUM)

**Location:** `core/jobs/message-processing-job.ts`

**Issue:** No visibility into queue health. Stuck jobs are not automatically detected.

**Evidence:**
- No queue monitoring dashboard
- No job timeout alerts
- No dead-letter queue for failed jobs

**Risk:** Messages silently drop if job processing hangs.

**Recommendations:**
- [ ] Add Bull UI for queue monitoring
- [ ] Implement job timeout (5 min default)
- [ ] Move failed jobs to DLQ with retry policy
- [ ] Alert on queue depth > 1000

---

### 8. Circular Import Risks (Impact: LOW-MEDIUM)

**Location:** `core/`, `services/`

**Issue:** Dependency injection sometimes creates circular dependencies (services import services).

**Risk:** Hard to trace module load order; potential runtime issues.

**Recommendations:**
- [ ] Use dependency injection container (tsyringe or manual registry)
- [ ] Lint for circular imports (`eslint-plugin-import`)
- [ ] Document module dependency order

---

**Recommendations:**

## Known limitations

### LLM Response Validation

LLM responses must strictly match `AgentLLMResponse` schema. If format is wrong:
- Response rejected, user sees error
- No graceful degradation
- No in-context learning to correct format

**Workaround:** Ensure prompts include JSON schema in system prompt.

### Enrichment Service Timeouts

Enrichment calls (TMDB, YouTube, etc.) have no individual timeout:
- If TMDB is slow, entire turn stalls
- Kernel can block for minutes

**Workaround:** Add 5-second timeout per enrichment call; fail gracefully.

### Memory Item Type Validation

Memory metadata is not validated against type:
- Saving movie with YouTube metadata is allowed
- Causes silent data corruption

**Workaround:** Add Zod schema per memory type; validate on save.

## Risk zones (high coupling)

```
🔴 core/model/credential-pool.ts
   ├─ Tightly coupled to all LLM providers
   ├─ No interface; direct imports from Gemini, Claude clients
   └─ Risky: Adding new provider requires code change

🔴 services/agent-orchestrator.ts
   ├─ Imports kernel, registries, enrichment, DB
   ├─ No clear boundaries
   └─ Risky: Hard to test; many mocks needed

🟡 core/kernel/hermes-kernel.ts
   ├─ Tightly coupled to tool registry
   ├─ Passes tools into LLM (schema coupling)
   └─ Risky: Tool schema changes break kernel

🟡 routes/conversations.ts
   ├─ Imports services, DB, enrichment
   ├─ No validation layer
   └─ Risky: Request validation scattered

🟢 core/enrichment/
   ├─ Clean interfaces (each service is independent)
   ├─ No inter-service coupling
   └─ Safe: Easy to test or mock

🟢 db/schema/
   ├─ No business logic; pure definitions
   └─ Safe: No dependencies
```

## Unresolved decisions

### BYOK Provider Key Storage (NEX-53)

**Question:** Should users provide their own AI API keys (Gemini, Claude)?

**Current state:** `provider_keys` table exists but unused; all users share Nexo's credentials.

**Implications:**
- If enabled: Major security review needed (key encryption, isolation)
- If disabled: Users locked into Nexo's provider choices

**Status:** Pending product decision.

---

### Skill System Extensibility (NEX-30)

**Question:** Should skills be user-definable (no-code or LLM-generated)?

**Current state:** Skills are stored in DB, but no UI to edit them.

**Implications:**
- Complex: Skills can execute arbitrary tools
- Risk: Malicious skills could cause harm
- Opportunity: User customization unlock

**Status:** Pending security & product review.
